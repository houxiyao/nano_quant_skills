# Rule Delivery Workflow

每一条“七看八问”规则都按同一个最小交付流程推进，避免一次做完 15 条导致口径失控。

## 单条规则必须回答的 6 个问题

1. 这条规则到底在判断什么业务问题？
2. 输出是数值、分档、通过/不通过，还是文字结论？
3. 需要哪些表和字段？哪些是直接字段，哪些是派生指标？
4. 时间窗口是什么？年报、TTM、单季度、滚动 N 年，还是日频区间？
5. 阈值如何定义？绝对阈值、行业分位，还是相对自身历史？
6. 如果数据缺失、口径冲突或样本过少，如何降级？

## 单条规则的交付步骤

### Step 1. 明确规则规格

- 记录规则编号，例如 `look-01` 或 `question-03`
- 记录规则标题
- 记录一句话业务意图
- 记录输入参数：股票代码、分析日期、窗口、阈值
- 记录输出 contract：字段名、单位、解释方式

### Step 2. 做数据映射

- 对照 `docs/` 和 `references/data-coverage.md`
- 标记每个输入项属于哪一类：`direct`、`derived`、`missing`
- 对 `derived` 指标写出明确公式
- 对 `missing` 项暂停编码并与用户讨论

### Step 3. 设计实现

- 优先使用 Python 脚本固化
- 把查询、计算、评分、解释拆开
- 明确日期对齐逻辑，尤其是 `ann_date`、`end_date`、`trade_date`
- 明确空值和异常值处理

### Step 4. 真实样本实测

- 至少跑 1 个真实股票样本
- 如果规则天然有正反案例，优先跑 2 个样本
- 输出原始证据、关键中间值、最终结论
- 如果实测结果违反直觉，先回查公式和日期口径，再决定是否调规则

### Step 5. Code Review

- 使用现有 code-review skill 做 review
- review 重点：
  - 公式是否和业务定义一致
  - 时间口径是否正确
  - 是否误用 `ann_date` 和 `end_date`
  - 空值、停牌、异常值是否妥善处理
  - 输出是否可解释

### Step 6. 回写注册表

更新 `assets/rule_registry.json` 中对应 rule：

- `title`
- `status`
- `parallel_lane`
- `tables`
- `derived_metrics`
- `missing_data`
- `script`
- `test_status`
- `review_status`
- `notes`

## Done 定义

一条规则只有同时满足以下条件，才算真正完成：

1. 规则规格已明确，没有关键口径空白。
2. 数据映射完成，缺口已经被消除或被明确接受。
3. Python 脚本已落地。
4. 至少一次真实样本实测通过。
5. 至少一次 code review 完成。
6. 规则注册表已更新。

## 不应该直接编码的情况

- 用户只给了规则名称，没有给定义
- 规则高度依赖行业对比，但行业基准尚未定义
- 需要 docs 之外的数据，且替代方案不明确
- 需要多重派生指标，但公式和窗口没有定
